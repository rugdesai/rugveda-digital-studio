## Two hats, one head

When I sit down to build, I try to hold two questions at once:

- **Who is this for, and what are they actually trying to do?**
- **What's the smallest, cleanest thing I can ship that answers that?**

The trap is picking one. Pure PM without code drifts into decks. Pure code without PM drifts into a beautifully-engineered thing nobody uses.

## A tiny checklist I use before writing code

1. One-sentence user problem. If I can't write it, I don't understand it yet.
2. The riskiest assumption. What breaks the whole thing if I'm wrong?
3. The smallest test. What's the least I can build to learn?

## Craft is a product feature

Latency, empty states, error copy, keyboard focus — users don't have words for these, but they feel them. Treat them as product decisions, not polish.
